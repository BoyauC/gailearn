# Next Word Prediction 遊戲開發規格

## 專案目的

本目錄是一個可直接部署到 GitHub Pages 的繁體中文互動式教學遊戲，主要對象為臺灣國中生。遊戲以「節點星空」呈現語言模型逐步預測下一個詞的過程，讓玩家理解：AI 生成內容是依上下文與統計權重進行預測，語句流暢或機率較高，不代表內容已經查證或必然正確。

所有在本目錄工作的 Codex 代理都必須先讀完本文件，再修改程式或資料。

## 不可偏離的教學原則

1. 不把 AI 擬人化為「知道」「相信」或「想起」史實；介面文字應使用「預測」「候選」「權重」「上下文關聯」。
2. 節點亮度、大小與百分比只表示模擬的相對機率，不表示正確性。
3. 每次選詞後，下一輪候選必須依目前已選路徑切換；不可四輪都顯示固定答案。
4. 高機率選項不應固定為正解或錯解，避免玩家用機率排序猜題。
5. 即使玩家生成符合史實的敘述，也要提醒「預測正確仍不等於完成查證」。
6. 結局必須並列「玩家／AI 路徑生成」與「查證後敘述」，並提供可信來源。
7. 初階關卡可使用明確事實錯誤；未來進階關卡應加入「語意合理但未經證實的推論」。
8. 所有機率皆為教學模擬值，除非未來確實串接可回傳詞元機率的模型；介面必須保留此註記。

## 目前實作範圍

- 預計最終有 3 個案例。
- 每次開始或接續遊玩時，從 `active=1` 的案例亂數抽取。
- 有兩個以上案例時，不得立刻重複上一題；只有一題時允許重玩。
- 目前只完成案例 `history_koxinga_1661`：鄭成功攻臺地點之謎。
- 玩家完成一題後可選擇「再玩一個案例」或「離開遊戲」。
- 「離開遊戲」回到本遊戲的封面畫面，不導向外部網站。

## 技術限制

- 使用原生 HTML、CSS、JavaScript，不引入框架、套件管理器或建置步驟。
- `index.html` 必須位於本目錄第一層。
- 所有路徑使用相對路徑，確保 GitHub Pages 子路徑部署正常。
- 不依賴後端、API 金鑰、資料庫或跨網域服務。
- 不使用內嵌 SVG 當裝飾。星空以 CSS 與 Canvas 繪製。
- GitHub Pages 與本機 HTTP server 必須可正常載入 CSV。瀏覽器直接以 `file://` 開啟時可能因安全政策阻擋 `fetch`，介面應顯示清楚的啟動提示，不可靜默失敗。
- 文字內容以 UTF-8 CSV 為主要資料來源；不要把案例敘述複製成 JavaScript 常數。

## CSV 編輯規則

- CSV 第一列為欄名，逗號分隔，文字若含逗號、換行或雙引號必須用雙引號包覆，內部雙引號寫成兩個雙引號。
- ID 僅使用英文字母、數字與底線；已發布 ID 不任意更名。
- 布林值使用 `1` 或 `0`。
- 機率使用整數百分比，同一 `case_id + group_id` 的三個節點基礎值總和必須為 100。程式會依玩家完整路徑做最多 2 個百分點的情境微調，畫面上仍須合計為 100。
- 每一個有效節點群組必須剛好有三個選項。
- `next_group` 在第 1～3 步不可留白；第 4 步必須留白。
- `segments.csv` 同一案例的 `order` 必須連續，拼接後應等於開場敘述。
- 案例文字的小幅修改優先改 CSV，不應為此修改 HTML 或 JavaScript。

### `cases.csv` 必要欄位

`case_id,active,category,difficulty,title,kicker,intro_instruction,story_prefix,story_suffix,reveal_hit,reveal_miss,prediction_prompt,verdict_question,verified_story,comparison_note,ending_wrong,ending_right,source_label,source_url,source_label_2,source_url_2,simulation_note`

其中 `story_prefix` 是填入生成詞之前的句子，`story_suffix` 是之後的句子。四個節點選出的 `token` 以空字串連接後置入兩者之間。

每個案例至少提供一筆查證文獻於 `source_label` 與 `source_url`；第二筆文獻填入 `source_label_2` 與 `source_url_2`。沒有第二筆時兩欄可留白，結算頁會自動隱藏空白項目。

### `segments.csv` 必要欄位

`case_id,order,text,suspicious,type,explanation`

`type` 目前允許 `fact`、`inference`、`context`。玩家可以複選；只要選中任一 `suspicious=1` 片段就算有注意到核心問題。

### `nodes.csv` 必要欄位

`case_id,group_id,step,option_order,token,label,probability,next_group,is_verified_direction,reason`

- `token`：加入生成句子的文字，可包含「的」等銜接字。
- `label`：星空節點顯示文字。
- `next_group`：選擇後下一輪讀取的群組。
- `is_verified_direction`：僅供結算解說，不可在選擇階段以顏色洩漏答案。
- `reason`：選擇後顯示的統計關聯旁白。

## 標準遊戲流程

1. **封面**：說明任務、案例數與教學模擬機率，玩家按「開始探索」。
2. **批判閱讀**：把 `segments.csv` 片段渲染成可點選按鈕，玩家標記需要查證的詞。
3. **揭示調查點**：依是否選中可疑詞顯示 `reveal_hit` 或 `reveal_miss`，但先不公布正解。
4. **節點星空**：連續四輪、每輪三選一。節點大小與光暈依機率變化，不能用正誤色彩提示。
5. **可信度判斷**：把四個 token 填回敘述，詢問是否可以直接視為史實。
6. **路徑回放與比對**：顯示四步機率軌跡、生成敘述、查證敘述、錯誤成因與來源。
7. **結束選擇**：再玩一題或離開；未來有多題時亂數且避免立即重複。

## 介面與視覺方向

- 氛圍：深藍夜空、青藍與金色微光，像在資料宇宙中追蹤詞語。
- 主要內容區不可只靠 Canvas；文字、按鈕、狀態都必須是可存取的 DOM 元素。
- 星空節點為圓角按鈕，大小差異需受上限控制，低機率選項仍需容易點擊。
- 路徑選中後留下光線或光點軌跡；減少動態模式下停用非必要動畫。
- **平板是主要設計基準，橫式是最佳體驗**：優先以 1024×768、1180×820、1366×1024 等橫式視窗安排資訊密度；直式平板必須完整支援，較寬的 PC 是延伸版面，手機是相容性版面。
- 橫式平板維持三個候選節點水平並列，讓重要上下文、三個節點與選擇後說明盡量同時可見。
- 直式平板可增加垂直間距或調整區塊順序，但保留相同功能、內容與三選一關係；旋轉裝置只應觸發 CSS 重新排版，不得重置目前案例、已標記片段或預測路徑。
- 桌機、平板與手機皆可操作；橫式平板使用 `768px–1366px + orientation: landscape`，直式平板使用 `621px–900px + orientation: portrait`，手機窄版斷點約 620px。
- 平板主要操作按鈕至少 56px 高，一般可點區至少 44×44px，正文至少 16px；需考慮瀏海與安全區 `env(safe-area-inset-*)`。
- 不以紅／綠單獨表示正誤，所有狀態同時提供文字。

## 平板操作原則

- 以單指點按為主要互動，不要求拖曳、滑鼠右鍵、雙擊、長按或精準游標操作。
- 不可把關鍵資訊或操作只放在 hover 狀態；觸控裝置的按壓狀態使用 `:active`，且不得造成節點明顯位移。
- 一個任務畫面只保留一個主要前進動作。按下節點後立即停用同組按鈕，避免連點造成重複路徑。
- 橫式優先壓縮裝飾性空白、頁首和卡片內距，不壓縮字級與觸控目標；預測畫面應盡量在 1024×768 同時呈現進度、上下文、三個節點及關聯說明。
- 直式平板利用較長的垂直空間安排內容，允許自然捲動，但主要前進按鈕不得被固定元素遮擋，也不得顯示「請旋轉裝置才能繼續」。
- 批判閱讀片段在兩種方向都需有足夠行高與至少 44px 點按高度，選取狀態同時使用底線、背景與 `aria-pressed`。
- PC 版保留鍵盤焦點與 hover 輔助效果，但不得因此改變平板版的資訊順序或主要操作流程。

## 無障礙需求

- 所有互動可用鍵盤完成，焦點樣式清楚。
- 畫面切換後將焦點移到新畫面的標題，並以 `aria-live` 宣告重要回饋。
- Canvas 標記為裝飾性，不承載唯一資訊。
- 節點按鈕的無障礙名稱包含候選詞與機率。
- 支援 `prefers-reduced-motion`。
- 色彩對比以 WCAG AA 為最低目標。

## 驗收清單

每次修改後至少確認：

1. 使用本機 HTTP server 開啟第一層 `index.html`，三份 CSV 均成功載入。
2. 封面 → 找詞 → 揭示 → 四輪選擇 → 可信度判斷 → 結算可完整走完。
3. 每輪剛好三個節點且機率總和為 100。
4. 選擇不同第一步會出現不同第二步候選。
5. 生成敘述確實由玩家四次選擇組成。
6. 結算顯示來源連結、教學模擬機率註記及兩段敘述比對。
7. 「再玩一個案例」與「離開遊戲」均可用。
8. 360px 寬度不出現水平捲動；鍵盤操作不會卡住。
9. JavaScript console 無錯誤；CSV 載入失敗時顯示可理解的錯誤畫面。
10. 以 1024×768、1180×820 與 1366×1024 橫式尺寸確認三個節點保持橫排、所有主要按鈕容易點按，且預測畫面的上下文與說明不被裁切。
11. 以 768×1024 與 820×1180 直式尺寸確認完整流程可用、允許合理捲動、不要求旋轉裝置，旋轉前後遊戲狀態不重置。
12. 以觸控／無 hover 條件確認所有功能可完成；再以 1440×900 PC 尺寸確認版面沒有被平板規則過度放大。

## 新增第二、三案例

新增案例時不複製頁面：只在三份 CSV 增加相同 `case_id` 的資料。每題必須包含一個根群組 `root`、四個步驟與完整查證來源。案例應先由可靠的一手或權威教育／典藏來源查核，再寫入 `verified_story`、`source_url`，並盡可能提供第二筆交叉查證文獻。若新案例需要新的遊戲機制，先維持舊 CSV 向後相容，再擴充欄位與解析器。

## 品牌與視覺規格

- 對外名稱固定使用「AI的預測與學習」，首頁主標題固定為「AI提供的，會是真的嗎？」。
- 色彩依 `遊戲風格顏色配置.pdf`：主背景 `#E6E8DD`、淺草綠 `#CCE1A0`、鼠尾草綠 `#A8D59C`、強調黃綠 `#B7D678`／`#ACD47D`、卡其 `#C9C391`、主文字 `#666057`、輔助橄欖色 `#818261`。
- 首頁以思思角色作為「下一個詞」的視覺引導；裝置畫面上的文字必須使用 HTML 疊加，避免圖片文字模糊並維持無障礙對比。
- 操作頁延續暖米色與綠色系；節點大小與亮度呈現模擬機率，但不得使用近似「正確答案」的成功／失敗色暗示。
- 頁面仍以平板橫式為主要設計基準，同時維持平板直式、手機與桌機可完整操作。
- 角色圖片位於 `assets/images/sisi-60427.png`，保留透明背景，不任意變形、裁切臉部或改變角色原有配色。

## 修改邊界

- 保留使用者自行修改過的案例文字，不因程式重構覆寫 CSV。
- 不更動本目錄以外的其他章節，除非使用者明確要求整合導覽。
- 不提交部署、推送 GitHub 或修改 repository settings，除非使用者明確要求。
